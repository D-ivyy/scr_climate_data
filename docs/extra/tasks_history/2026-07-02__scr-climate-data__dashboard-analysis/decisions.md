# Decisions: SCR Climate Data Dashboard and Output Analysis

## 1. Use `assetName` As The Join-Back Key

### Decision

Use returned `Output.assetName` to join SCR results back to the private manifest:

```text
Output.assetName
  -> scr_manifest.csv.scr_asset_name
  -> plant_uuid / tenant asset identity
```

Treat `assetId` as vendor metadata only.

### Rationale

`assetId` is assigned/returned by SCR and is not stable enough to be the InfraSure canonical key. `assetName` is generated by our upload process, so it can encode a stable internal identifier while still being acceptable to SCR. This also works for both reference plants and future tenant/greenfield assets.

## 2. Start With Plant-Level Physical Uploads

### Decision

Use plant-level assets for V1 SCR uploads.

### Rationale

The raw EIA location source currently provides latitude/longitude at plant level, not reliable generator-level coordinates. Generator-level modeling remains a future idea for hybrid or multi-site assets once more granular coordinates are available.

## 3. Keep Manifest Separate From SCR Upload

### Decision

Generate separate outputs for:

- upload file,
- private manifest,
- reject diagnostics.

### Rationale

SCR does not need internal InfraSure metadata such as plant UUIDs, portfolio names, or operating-entity detail in the upload file. The manifest preserves join-back without exposing unnecessary internal/customer information.

## 4. Treat Operating Entity As Optional For V1 Upload Tests

### Decision

Do not include operating entity in the initial compact upload when it is not required for SCR acceptance.

### Rationale

The initial goal is validating SCR upload acceptance and result return. Avoiding unnecessary operating entity exposure reduces privacy and commercial risk while testing the workflow.

## 5. Preserve Raw SCR Numeric Values

### Decision

Keep raw SCR model-output values in JSON and display percent-style or basis-point views only as readability transforms.

### Rationale

Impact fields such as `adjustedTotalValueImpact` and `adjustedSubriskRevenueImpact` are small raw numeric values. Display transforms help humans read them, but SCR has not yet confirmed final product-facing unit labels.

## 6. Distinguish Physical Disruption From Transition Revenue Impact

### Decision

Physical dashboard views now distinguish:

```text
Physical value impact      -> adjustedTotalValueImpact
Physical disruption        -> adjustedTotalDisruption
Transition revenue impact  -> adjustedSubriskRevenueImpact
```

### Rationale

Physical disruption is a physical workbook metric tied to business continuity/revenue exposure. Transition revenue impact is a transition workbook metric tied to subrisks such as direct carbon cost and market demand shifts. They should not be merged or labeled as the same thing.

## 7. Recompute Transition Rankings When Filtering Drivers

### Decision

The transition driver selector recomputes rankings from row-level subrisk records when filtering to direct carbon cost or market demand shifts.

### Rationale

Filtering only the display table would create a mismatch between ranking, trend, and details. Recomputing from `transition.subrisks` keeps all transition views consistent.

## 8. Make Hazard Rows Expandable

### Decision

The hazard ranking row stays compact by default, and long indicator details live inside an expandable row.

### Rationale

Long labels such as `Fluvial & Pluvial Flood depth - 500 years RP` made the main ranking row unreadable. The expandable pattern preserves scanability while keeping detailed evidence one click away.

## 9. Show Returned Hazard Curves Separately From Derived Response Curves

### Decision

Use two separate sections inside opened hazard rows:

- returned hazard curves: horizon on x-axis,
- magnitude response: indicator magnitude on x-axis.

### Rationale

Returned hazard curves are direct visualizations of SCR fields by horizon. Magnitude-response plots are derived joins between `indicatorValue` and hazard-level metrics; they are useful for exploration but should not be presented as official SCR vulnerability functions.

## 10. Mark Flat Returned Data As Flat

### Decision

Treat near-identical values as flat in mini charts to avoid visual spikes caused by tiny floating-point differences.

### Rationale

Flood values in the raw workbook are effectively flat across future horizons, but tiny precision differences can visually exaggerate movement if autoscaled too aggressively. The dashboard should not imply movement where SCR returned none.

## 11. Keep The Dashboard Static For V1

### Decision

Use plain HTML/CSS/JavaScript plus a Python data builder instead of a framework or Streamlit.

### Rationale

The V1 dashboard is a local analysis tool, not a platform feature. Static assets make it simple to run, easy to commit, and easy to inspect without npm/build dependencies.
